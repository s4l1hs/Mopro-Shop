# PD-07 — review reviewer-name + photos (discovery)

> Surfaces the reviewer's (masked) name + attached photos in the PDP reviews, per
> #158's PD-07 deferral plan. §5-safe (identity + attachments **service** calls,
> never a cross-schema JOIN). Catalog-backend + PDP-reviews-UI lane.

## Current read path

- `GET /products/{id}/reviews` → `handleProductReviews` (`cmd/core-svc/home_handlers.go`)
  builds `reviewJSON` `{id, userId, rating, title, body, helpfulCount,
  votedByCurrentUser, createdAt}` from `svc.ListReviews` + `svc.ReviewsSummary`.
  **No reviewer name, no photos.**

## ⚠ Discovery shift — the reviews endpoint is NOT in the OpenAPI spec

`grep reviews api/openapi.yaml` → nothing (only a `rating_avg` description). The
mobile reviews path is **hand-written, raw-Dio** (`reviews_provider.dart`: `_dio`
+ `/products/$id/reviews`, a hand-parsed camelCase `Review` model) — by design,
not the generated client. So **PD-07 is NOT a codegen change**: add the fields to
the Go `reviewJSON` + the hand-written Dart `Review` model + the UI. No
`openapi.yaml` edit, no `make api-gen`, no Dart fan-out, no gen-sync trigger. The
live-handler contract test asserts the handler output **directly** (field
presence + shape), the same way #158 guarded `image_urls` (it can't VisitJSON a
schema that doesn't exist).

## §5-safe metadata sources (in-process service calls)

- **Reviewer name** — `product_reviews.user_id` is a soft ref to `identity_schema`.
  Resolve via **`identity.Service.GetUser(ctx, userID)`** → `User.Name` (full
  display name, not encrypted). **Mask Trendyol-style** ("Ahmet Yılmaz" → "A** Y**").
  On error / empty → empty string; the UI shows an i18n fallback.
- **Photos** — `attachments_schema.photo_attachments` (`entity_type='review'`).
  Resolve via **`attachments.Service.ListByEntity(ctx, "review", reviewID)`** →
  `[]PhotoAttachment{StorageKey}`; CDN-map each via **`mediaurl.CDNUrl`** (the
  PD-06 helper). Always a non-nil `[]` (never null).
- Both services are already constructed in `main.go` (`identitySvc` :386,
  `attachmentsSvc` :182/194) and in scope at the `handleProductReviews` wiring
  (:518). Narrow handler interfaces (à la #158's `deliveryEstimator`) keep deps tight.
- **Fan-out:** N reviews/page (≤50) → ≤50 `GetUser` + ≤50 `ListByEntity` calls.
  Acceptable at launch scale; batch is a later optimization (note it).

## Plan (one PR, commit per concern)

1. **Backend:** `reviewJSON` += `reviewerName` + `photoUrls`; `handleProductReviews`
   gains the two service deps + per-review resolution (masked name, CDN photos);
   `main.go` wiring. A `maskName` helper.
2. **Mobile:** the hand-written `Review` model += `reviewerName`/`photoUrls`
   (`fromJson`); `review_row.dart` renders the name header + a photo-thumbnail
   strip (lightbox optional); `reviews.anonymous` i18n fallback.
3. **Contract test:** extend the cmd/core-svc live-handler test — call
   `handleProductReviews` with stubs (a review + a faked user + a faked photo) and
   assert `reviewerName` is masked + `photoUrls` are CDN-mapped + non-null.

## §4 split-bailout

If the attachments path proves heavy, ship reviewer-name first, photos second —
**never** a cross-schema JOIN to shortcut it.
